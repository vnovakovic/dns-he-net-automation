## 2. Windows Service Error 1067 — TLS Certificate Fix

### Root cause
The Windows installer bundled `.env.example` which sets `SSL_CERT` and `SSL_KEY` to paths under
`C:\Program Files\dnshenet-server\`. Those files were gitignored and never present in CI builds.
On first service start, `ListenAndServeTLS` failed with "no such file or directory" → Error 1067.

### Diagnosis
`C:\dnshenet-early.log` showed:
```
[...] browser launched OK
[...] main() exiting            ← crash after browser launch, before HTTP bind
```
This pinpointed the failure to the `ListenAndServeTLS` call, not the browser.

### Fix: `gen-cert` subcommand
Added a `gen-cert` CLI subcommand to `cmd/server/main.go`:
```
dnshenet-server.exe gen-cert --cert <path.crt> --key <path.key>
```
Generates an ECDSA-P256 self-signed certificate with:
- SAN: `localhost`, `127.0.0.1`, `::1`
- Validity: 10 years
- PEM output at the specified paths

Added to `installer/dnshenet-server.iss` `[Run]` section:
```pascal
Filename: "{app}\dnshenet-server.exe";
  Parameters: "gen-cert --cert ""{app}\server.crt"" --key ""{app}\server.key""";
  WorkingDir: "{app}";
  StatusMsg: "Generating self-signed TLS certificate...";
  Flags: waituntilterminated;
  Check: not FileExists(ExpandConstant('{app}\server.crt'))
```
The `Check:` function makes this idempotent — cert is only generated if it doesn't already exist.

## 4. Docker TLS Auto-generation

### Goal
Docker containers have no installer and no pre-start hook. Without a cert on disk,
`ListenAndServeTLS` fails and the container exits. The cert must be generated automatically
on first start when `SSL_CERT`/`SSL_KEY` paths are configured but the files are absent.

### Implementation

**`cmd/server/main.go`**

Refactored `runGenCert` into two functions:
- `genCertFiles(certPath, keyPath string) error` — pure cert generation, returns error (no `os.Exit`)
- `runGenCert(args []string)` — CLI wrapper, calls `genCertFiles`, exits on error

Added auto-gen block in `main()` after config load, before TLS setup:
```go
if cfg.SSLCert != "" && cfg.SSLKey != "" {
    _, certMissing := os.Stat(cfg.SSLCert)
    _, keyMissing := os.Stat(cfg.SSLKey)
    if os.IsNotExist(certMissing) || os.IsNotExist(keyMissing) {
        slog.Info("TLS cert/key not found — auto-generating self-signed certificate", ...)
        if err := genCertFiles(cfg.SSLCert, cfg.SSLKey); err != nil {
            slog.Error("failed to auto-generate TLS certificate", "error", err)
            os.Exit(1)
        }
    }
}
```

**Behaviour:**
- `SSL_CERT`/`SSL_KEY` empty → plain HTTP (reverse proxy deployments) — unchanged
- `SSL_CERT`/`SSL_KEY` set, files absent → auto-generate cert → HTTPS
- `SSL_CERT`/`SSL_KEY` set, files present → skip generation → HTTPS
- Generation failure → `os.Exit(1)` — no silent HTTP fallback

**`Dockerfile`**

```dockerfile
# Create /data directory owned by non-root service user
RUN mkdir -p /data && chown server:server /data

# Default TLS and DB paths inside the persistent volume
ENV SSL_CERT=/data/server.crt
ENV SSL_KEY=/data/server.key
ENV DB_PATH=/data/dnshenet-server.db

VOLUME ["/data"]
EXPOSE 9001
```

The `/data` volume holds the database and the auto-generated TLS certificate.
Without a named volume mount, the cert and DB survive container restarts but
are lost on `docker pull` + container recreation.

### Docker deployment example
```bash
docker run -d \
  -v dnshenet-data:/data \
  -e JWT_SECRET=<32-char-secret> \
  -e ADMIN_USERNAME=admin \
  -e ADMIN_PASSWORD=<strong-password> \
  -p 9001:9001 \
  harbor.e2.network/provision/dns-he-net-automation:latest
```

First start: cert auto-generated at `/data/server.crt` + `/data/server.key`.
Subsequent starts: cert already exists, skipped.

### Docker Playwright driver — three-attempt debugging

| Attempt | Error | Root cause | Fix |
|---------|-------|-----------|-----|
| 1 | `mkdir /home/server: permission denied` | No `PLAYWRIGHT_DRIVER_PATH` → playwright-go tried to create driver dir under `/home/server` (no-create-home) | Created `/ms-playwright-driver` owned by `server` user + `ENV PLAYWRIGHT_DRIVER_PATH` |
| 2 | `please install the driver first` | `PLAYWRIGHT_DRIVER_PATH` set *after* `playwright install` → driver wrote to `/root/.cache` during build, `/ms-playwright-driver` empty at runtime | Moved both `ENV` vars *before* `playwright install` |
| 3 | Success ✓ | | |

Final Dockerfile order:
```dockerfile
ENV PLAYWRIGHT_BROWSERS_PATH=/ms-playwright       # must be before install
ENV PLAYWRIGHT_DRIVER_PATH=/ms-playwright-driver  # must be before install
RUN playwright install --with-deps chromium        # writes to both paths
RUN chown -R server:server /ms-playwright-driver   # write access for runtime lock files
RUN chmod -R a+rX /ms-playwright                   # read access for server user
```

## Known Limitations / Follow-up

- The self-signed cert has `localhost` / `127.0.0.1` / `::1` SANs only. For deployments
  where the service is accessed by hostname or external IP, the cert will fail browser
  TLS validation. Production deployments should replace it with a CA-signed cert or
  configure a reverse proxy (nginx/Traefik) for TLS termination.
- The GitLab API token used during this session (`glpat-xC7xz3sCzJBMxNZqhxLY`) has
  expired and needs to be regenerated if API automation is needed again.
