**Status:** Complete

## Goal

Fix four bugs discovered during live testing:
1. `UpdateRecord` always returned `browser_error` (30s timeout)
2. Record rows could not be clicked for editing
3. Browser errors were invisible in server logs
4. First user registration on the admin Users page silently failed

---

## Root Causes and Fixes

### Bug 1 — `EditExistingRecord` used invalid CSS ID selector
**File:** `internal/browser/pages/recordform.go`

HE.net record row IDs are purely numeric (e.g. `8900607586`). CSS ID selectors
(`tr#8900607586`) are invalid — the CSS spec forbids leading digits. Playwright
threw a silent parse error and returned no match.

**Same root cause** as the `ParseRecordRow` bug fixed in the previous session
(zonelist.go). Both required the same attribute-selector fix.

```go
// Before (broken):
rowSelector := fmt.Sprintf("tr#%s", recordID)

// After (fixed):
// WHY attribute selector: HE.net record row IDs are purely numeric. CSS ID
// selectors reject leading digits. tr[id="..."] has no such restriction.
rowSelector := fmt.Sprintf(`tr[id="%s"]`, recordID)
```

---

### Bug 2 — `SelectorRecordSubmit` only matched the add form, never the edit form
**File:** `internal/browser/pages/selectors.go`

`input#_hds` is reused for both add and edit operations, but dns.he.net changes
its `value` attribute at runtime via JavaScript:
- Add mode:  `value="Submit"`
- Edit mode: `value="Update"`

The old selector `input[name="hosted_dns_editrecord"][value="Submit"]` never
matched during edit, causing a 30-second Playwright timeout on **every**
`UpdateRecord` call.

**Discovery method:** Navigated to live dns.he.net via Playwright MCP, clicked
a record row, inspected the DOM via `page.evaluate()`:

```json
{
  "submitInput": "<input id=\"_hds\" name=\"hosted_dns_editrecord\" value=\"Update\" type=\"submit\">"
}
```

```go
// Before (broken for edit):
SelectorRecordSubmit = `input[name="hosted_dns_editrecord"][value="Submit"]`

// After (fixed — matches both add and edit):
// WHY no [value="Submit"]: add mode uses "Submit", edit mode uses "Update".
// Matching by name alone is safe — only one such input exists on the page.
SelectorRecordSubmit = `input[name="hosted_dns_editrecord"]`
```

### Bug 3 — `handleBrowserError` swallowed the actual error silently
**File:** `internal/api/handlers/records.go`, `bind.go`, `sync.go`

`handleBrowserError(w, err)` wrote the JSON response but never logged the error.
All browser failures were invisible in the server log — only `browser_error` JSON
appeared in the response body. This made bugs 1 and 2 extremely hard to diagnose.

**Fix:** Added `slog.ErrorContext` and changed signature to accept `*http.Request`:

```go
func handleBrowserError(w http.ResponseWriter, r *http.Request, err error) {
    slog.ErrorContext(r.Context(), "browser operation failed", "error", err)
    // ... status code mapping unchanged
}
```

All callers in `records.go`, `bind.go`, and `sync.go` updated to pass `r`.

### Bug 4 — Admin user registration silently failed on empty user list
**File:** `internal/api/admin/templates/users.templ`

The Register form targeted `hx-target="#users-table-body"` but the `<tbody>` with
that ID was only rendered when `len(users) > 0`. On an empty list, the template
rendered a plain `<p>` message instead of a table — htmx found no target element
and silently dropped the response.

The user **was** created in the DB (confirmed via curl with Basic Auth), but
nothing appeared in the UI.

**Fix:** Always render the table and `<tbody id="users-table-body">`. Move the
empty-state message inside a `<tr>` for valid table markup:

```templ
// Before: conditional table — tbody missing when empty
if len(users) == 0 {
    <p class="text-muted">No account users registered yet.</p>
} else {
    <table>...<tbody id="users-table-body">...</tbody></table>
}

// After: table always present — htmx always has a target
<table class="data-table">
    <thead>...</thead>
    <tbody id="users-table-body">
        if len(users) == 0 {
            <tr><td colspan="3" class="text-muted">No account users registered yet.</td></tr>
        } else {
            for _, u := range users { @UserTableRow(u) }
        }
    </tbody>
</table>
```
